Safari VT Edge-to-Edge Repro

Bug: On Safari (iOS 18.2+) with edge-to-edge displays, the View Transitions root snapshot renders the floating header/footer as a solid color (white) during the transition, instead of transparent.

Each variant below tests a different strategy to suppress or work around the root snapshot while keeping the shared-element card transition working.

Variant 1
Baseline (reproduces the bug)
Standard View Transition with shared card element. Root crossfade is left at defaults — the header/footer flash white during transition.
/* No overrides — default VT behavior */
Variant 2
No Root Animation
Disable root snapshot animation entirely. The old root still exists but doesn't animate — shared element should still morph.
::view-transition-old(root), ::view-transition-new(root) { animation: none; }
Variant 3
Hide Root Old Snapshot
Set display:none on the old root snapshot only. The new state appears immediately; the card still crossfades as a shared element.
::view-transition-old(root) { display: none; }
Variant 4
Hide Root Both (Old + New)
Hide both old and new root pseudo-elements. Only the shared element group remains visible during the transition.
::view-transition-old(root), ::view-transition-new(root) { display: none; }
Variant 5
Root Old Opacity 0
Keep the root animation timing but set old root opacity to 0 — the snapshot is captured but never made visible.
::view-transition-old(root) { opacity: 0 !important; }
Variant 6
Page Wrapper as Shared Element
Give the page background wrapper its own view-transition-name so it's also an animated shared element, not a static snapshot.
.page-bg { view-transition-name: page-bg; } ::view-transition-old(root) { animation: none; }
Variant 7
Types-Scoped Transitions
Use startViewTransition({types: ['card-expand']}) with :active-view-transition-type() to scope CSS only when this specific transition runs.
html:active-view-transition-type(card-expand) ::view-transition-old(root) { animation: none; }
Variant 8
Clip Root to Exclude Safe Areas
Apply clip-path to the root group to exclude the safe-area-inset regions. If the root snapshot is what paints solid white in those areas, clipping it out might let the real page show through.
::view-transition-group(root) { clip-path: inset( env(safe-area-inset-top, 0px) 0 env(safe-area-inset-bottom, 0px) 0 ); }
Variant 9
BG: Transparent on Root Old/New
Set background: transparent on ::view-transition-old(root) and ::view-transition-new(root). Safari may sample these backgrounds for the header/footer bar color.
::view-transition-old(root), ::view-transition-new(root) { background: transparent; }
Variant 10
BG: Transparent on Root Group
Set background: transparent on ::view-transition-group(root) — the container that wraps the image pair and covers the full viewport.
::view-transition-group(root) { background: transparent; }
Variant 11
BG: Transparent on ::view-transition Overlay
Set background: transparent on the top-level ::view-transition pseudo-element — the outermost VT overlay covering the entire viewport.
::view-transition { background: transparent; }
Variant 12
BG: Transparent on ALL Root VT Elements
Aggressively set background: transparent on every root VT pseudo-element: overlay, group, image-pair, old, and new.
::view-transition, ::view-transition-group(root), ::view-transition-image-pair(root), ::view-transition-old(root), ::view-transition-new(root) { background: transparent; }
Variant 13
Combo: Transparent BG + No Root Animation
Combines variants 2 + 9: root old/new get both background: transparent and animation: none.
::view-transition-old(root), ::view-transition-new(root) { background: transparent; animation: none; }
Variant 14
Combo: Transparent BG + Hide Root Old
Combines variants 3 + 9: old root hidden with transparent bg, new root also transparent as a safety net.
::view-transition-old(root) { background: transparent; display: none; } ::view-transition-new(root) { background: transparent; }
Variant 15
Structural: Transparent HTML/Body
html and body have background: transparent always. White lives only on the .page wrapper. Safe areas outside the wrapper are genuinely transparent — not white.
html, body { background: transparent; } .page { background: white; }
Variant 16
JS: Transparent During VT Callback
Inside startViewTransition() callback, set html/body to transparent via JS. The new-state snapshot captures them as transparent. Restored after transition finishes.
// In startViewTransition callback: document.documentElement.style.background = 'transparent'; document.body.style.background = 'transparent'; // After transition.finished: restore
Variant 17
Combo: Structural Transparent + VT Overlay Transparent
Combines variant 15 (structural transparent html/body) with variant 12 (transparent bg on all VT pseudo-elements). Double-layered CSS-only approach.
html, body { background: transparent; } .page { background: white; } ::view-transition, ::view-transition-group(root), ::view-transition-image-pair(root), ::view-transition-old(root), ::view-transition-new(root) { background: transparent; }
Variant 18
JS: Aggressive Transparent (html + body + .page)
Inside VT callback, set html, body, AND .page all to transparent. The entire viewport has no opaque white at snapshot time. Maximum transparency.
// In startViewTransition callback: document.documentElement.style.background = 'transparent'; document.body.style.background = 'transparent'; pageEl.style.background = 'transparent'; // After transition.finished: restore all
Variant 19
Zero White — No White Anywhere
html, body, and .page all transparent. Every VT pseudo-element transparent + animation: none. The only non-transparent element is the card itself. If Safari still flashes white, it's not from CSS.
html, body { background: transparent; } .page { /* no background */ } ::view-transition, ::view-transition-group(root), ::view-transition-image-pair(root), ::view-transition-old(root), ::view-transition-new(root) { background: transparent; animation: none; }
Variant 20 🔬
Color Trace — Every Element Distinct Color
Each element (html, body, .page, every VT pseudo-element) gets a unique bright color. VT slowed to 2s. Match the safe-area flash color to the legend to identify the exact source element.
html → 🔴 RED (#FF1744) body → 🟠 ORANGE (#FF9100) .page → 🟡 YELLOW (#FFEA00) ::view-transition → 🟣 MAGENTA ::view-transition-group → 🟢 GREEN ::view-transition-old → 🔵 BLUE ::view-transition-new → 🩷 PINK ⬜ WHITE = Safari default/system
Variant 21
Body = BLACK — Confirm Control
V20 proved Safari samples body's background. This sets body { background: #000 }. If safe areas turn black, we confirm full control — the fix is just finding the right color value.
body { background: #000000; }
Variant 22
Body Background: none
Uses background: none instead of background: transparent. The none shorthand resets all background properties to initial. Safari's compositing might handle this differently.
body { background: none; } ::view-transition, ::view-transition-group(root), ::view-transition-old(root), ::view-transition-new(root) { background: transparent; }
Variant 23
Body = Canvas (System Color)
Sets body { background: Canvas } — the CSS system color keyword for the OS's native background. Might blend with the system bars seamlessly in both light and dark mode.
body { background: Canvas; }
Variant 24
HTML = RED / Body = TRANSPARENT
Tests whether Safari resolves transparent body through the stacking context to html's RED, or hardcodes transparent→white. If safe areas flash RED, the computed background is used. If WHITE, transparent is special-cased.
html { background: #FF1744; } /* RED */ body { background: transparent; } /* RED flash = resolved through stacking WHITE flash = transparent hardcoded to white */
Variant 25
Pre-VT Transparent Body
Sets body transparent BEFORE startViewTransition() (not in callback). OLD snapshot also captures transparent body. Unlike V16 which only affected the NEW snapshot. Restored after transition.
// BEFORE startViewTransition: document.body.style.background = 'transparent'; const t = document.startViewTransition(() => { updateCardState(); // body still transparent }); t.finished.then(() => { document.body.style.background = 'white'; });
Variant 26
Root view-transition-name: none
Overrides the UA stylesheet's :root { view-transition-name: root; } with none !important. Tests whether the root can be excluded as a named VT element, preventing its snapshot from participating.
:root, html { view-transition-name: none !important; }
Variant 27
::view-transition NOT fixed
Overrides position: fixed on the top-level ::view-transition overlay. Sets position: relative + safe-area margins to pull it away from edges. WILL break visuals — tests if safe areas stop flashing when the overlay doesn't cover them.
::view-transition { position: relative !important; margin-top: env(safe-area-inset-top) !important; margin-bottom: env(safe-area-inset-bottom) !important; overflow: hidden !important; }
Variant 28
::view-transition-group(root) NOT fixed
Only the root GROUP is repositioned (position: absolute + inset top/height calc) away from safe areas. Overlay left alone. Isolates whether the group specifically fills safe areas.
::view-transition-group(root) { position: absolute !important; top: env(safe-area-inset-top) !important; width: 100vw !important; height: calc(100vh - env(safe-area-inset-top) - env(safe-area-inset-bottom)) !important; overflow: hidden !important; }
Variant 29
VT Overlay: Bottom Half Only (visible bounds)
Overlay positioned at bottom 50vh, 70% width. Does NOT cover Dynamic Island (top) or side edges. Magenta dashed outline shows exact bounds. Tests whether safe areas only flash where the overlay actually covers.
::view-transition { position: absolute !important; bottom: 0 !important; left: 15% !important; width: 70% !important; height: 50vh !important; outline: 4px dashed magenta; }
🔬 Diagnostic
Safe-Area Inspector — getComputedStyle + elementFromPoint
Runs getComputedStyle(documentElement, '::view-transition').position, probes elementFromPoint at top/bottom edges during VT, reports UA + iOS version. Tells you exactly what opaque layer sits in the safe-area bands.
// Pre-check: ::view-transition position // VT callback: elementFromPoint(w/2, 5) top // elementFromPoint(w/2, h-5) bottom // Post-VT: restore check
Variant 30
:root::view-transition { position: absolute }
Claude's selector fix: uses :root::view-transition instead of bare ::view-transition. The :root prefix matches UA specificity. Pulls overlay off fixed sampling path.
:root::view-transition { position: absolute !important; }
Variant 31
Toggle Fixed/Sticky → Absolute During VT
Before VT callback, finds all position: fixed / sticky elements in DOM, converts to absolute. Restored in .finished.finally(). Neutralizes any fixed element contributing to tint.
// Before startViewTransition: document.querySelectorAll('*').forEach(el => { if (getComputedStyle(el).position === 'fixed') el.style.position = 'absolute'; }); // .finished.finally(): restore
Variant 32
Group + Image-Pair Both Transparent
Sets background: transparent !important on ::view-transition-group(root), ::view-transition-image-pair(root), AND old/new. Previously tested pieces but not this exact combo.
::view-transition-group(root), ::view-transition-image-pair(root), ::view-transition-old(root), ::view-transition-new(root) { background: transparent !important; }
Variant 33
Root NOT a VT Participant
:root, html, body, .page { view-transition-name: none !important }. Only the card participates. If the root isn't captured as a VT element, Safari may skip the canvas fill entirely.
:root, html, body, .page { view-transition-name: none !important; } /* Only .card has view-transition-name */
Variant 34 ✅
No View Transitions API — CSS Only
Completely avoids VT: no view-transition-name, no startViewTransition(). Pure CSS transition: max-height. No VT overlay = no canvas fill = safe areas stay transparent. Card still expands smoothly.
/* No View Transitions API at all */ .card-content { transition: max-height 0.4s ease-out; } // JS: just toggle class, no startViewTransition
Testing note: Open each variant on Safari (iOS 18.2+) on a notched iPhone (14 Pro or later). Tap the page and observe the Dynamic Island / home indicator regions during the transition animation. The goal is for those areas to remain transparent (showing the system background) rather than flashing solid white.
